Закрыты все блокеры бренда честности из `AUDIT_CASCADE.md` + добавлена защита от DoS/взлома.
Движок и оплата не сломаны: **15/15 unit + 20/20 стресс PASS**, платёжный гейт цел.

## Что сделано

### Честность лендинга/страниц (блокеры Cascade #1, #3, #5)
- **Убран фейковый план Pro $29/mo** (лендинг EN+RU). Остались 2 реальных тарифа: Free + «Полный отчёт».
- **Цена сведена к реальной — $15** (бэкенд `PRICE_USD=15`). Раньше лендинг обещал $9, а списывало $15.
- **Убраны несуществующие фичи:** «PDF export», «version history», «downloadable certified report with seal», «shareable certificate». Осталось то, что реально есть: полный отчёт на экране + кнопка «Share verdict».
- **Убран email-фантом** (форма сохраняла email только в localStorage, никуда не уходил) — лендинг EN+RU + мёртвый CSS/JS.
- **Арбуз** (тест-кнопка фейкового ввода) **удалён из боевого** `validate_ru.html`. В dev-тренажёре `~/honest-validator` — остаётся для наших прогонов.

### Честность метода (Cascade #2, #5)
- В вердикт **REAL** добавлена метка границ: «checks historical robustness only — not hidden market-beta, capacity, or crisis coverage». Видна и в бесплатном вердикте.
- В отчёт добавлено поле `limits` (return-only анализ не видит скрытую бету/хвост/ёмкость/покрытие кризиса).

### Защита от DoS / взлома (Cascade #2 безопасность, запрос владельца)
- **Лимит тела запроса ≤ 1 MB** — гигантские POST режутся с 413, в память не читаются.
- **Лимит длины ряда ≤ 10 000 точек** + `n_trials` зажат в [1..1000].
- **Rate-limit по IP:** 30 запросов/мин на `/api/validate` и `/api/checkout` → 429. (IPN и unlock не лимитируем: HMAC/дёшево.) За nginx/Cloudflare реальный IP берётся из `X-Forwarded-For`.
- **`/api/report` закрыт fail-closed:** без заданного `REPORT_KEY` ручка отдаёт 403 (раньше бизнес-метрики — выручка/продажи — текли в интернет кому угодно). Ключ принимается из заголовка `X-Report-Key` или `?key=`, сравнение timing-safe.

### Мост «сессия → владелец» (запрос владельца)
- Добавлен `GET /api/ping?msg=...` — любая рабочая сессия шлёт владельцу строку в Telegram через сервер (например «Приходи к ПК»). Защита: тот же `REPORT_KEY` (fail-closed) + rate-limit. Пример:
  `curl -H "X-Report-Key: <ключ>" "https://isitalpha.com/api/ping?msg=Приходи+к+ПК"`

### Не тронуто (проверено)
- Платёжка: `np_ipn_valid` (HMAC-SHA512), `np_create_invoice`, `is_paid`, гейт free/paid — без изменений.
- Секреты только в `bootstrap.env` (в .gitignore). Бот не выводит средства.

## Чеклист деплоя (владелец)
1. **GitHub Desktop** → Commit всех изменений `isitalpha-app` → **Push**.
2. На сервере (VNC-терминал):
   ```
   cd /opt/isitalpha && git pull
   bash deploy/settg.sh          # впишет REPORT_KEY, рестартнёт, пришлёт тест-сводку в Telegram
   ```
   `settg.sh` теперь сам генерит `REPORT_KEY` — руками ничего вводить не надо.
3. Проверить: открыть https://isitalpha.com — цена $15, планов два, email-формы нет.
4. Если сводку в Telegram шлёшь вручную/кроном — добавь заголовок:
   `curl -H "X-Report-Key: <ключ из bootstrap.env>" 'https://isitalpha.com/api/report?send=1'`

## Рекомендации по защите (инфра — не код, включить в панелях)
- **Cloudflare (домен уже там):** включить проксирование (оранжевое облако) → это и есть основной анти-DDoS. При атаке — режим «Under Attack». Добавить Rate Limiting Rule на `/api/*`.
- **Бэкап состояния:** `paid_orders.txt` (кто оплатил) и `metrics.json` (счётчики) живут только на сервере. Раз в день копировать в надёжное место — иначе при потере сервера оплаченные клиенты не смогут пере-открыть отчёт. Сейчас продаж 0 — не срочно, но заложить.
- **REPORT_KEY** держать в секрете (он открывает бизнес-сводку).
